2. The Investigation Playbook

14 checks to run when a merchant or user is suspected of wrongdoing. Each shows the question, the SQL, and what to look for.

Check 1 — Are they trading with themselves on-chain?

Wash trading
SQL 
SELECT
  CASE WHEN creator_pubkey=$WALLET THEN counterparty_pubkey ELSE creator_pubkey END AS counterparty,
  COUNT(*) AS trades, SUM(amount::numeric/1000000) AS total_usd
FROM v2_trades
WHERE creator_pubkey=$WALLET OR counterparty_pubkey=$WALLET
GROUP BY 1 ORDER BY trades DESC;
Red flag Top counterparty has >50% of trades, or counterparty wallet resolves to any account linked by device/IP to the subject.
Then resolve wallet → account 
SELECT 'merchant' AS type, id, email FROM merchants WHERE wallet_address=$CP
UNION ALL SELECT 'user', id, email FROM users WHERE wallet_address=$CP;

Check 2 — How many accounts share their device?

Multi-account
SQL 
SELECT du.device_id, COUNT(DISTINCT du.entity_id) AS unique_accounts
FROM device_users du
WHERE du.device_id IN (SELECT device_id FROM device_users WHERE entity_id=$ID)
GROUP BY du.device_id ORDER BY 2 DESC;
Red flag Any device with >2 accounts — especially mixed merchant + user types — is suspicious.

Check 3 — Who else logged in from their IP(s)?

IP clustering
SQL 
SELECT DISTINCT entity_id, entity_type, MIN(created_at), MAX(created_at)
FROM ip_logs
WHERE ip IN (SELECT ip FROM ip_logs WHERE entity_id=$ID)
GROUP BY entity_id, entity_type
ORDER BY MIN(created_at);
Red flag >3 unique accounts from same residential / mobile IP within a short window. Note: shared corporate / café / VPN IPs are false positives — interpret with care.

Check 4 — Are there email-pattern siblings?

Email farm
SQL 
-- Strip digits/suffix to find pattern variants
SELECT id, email FROM merchants WHERE email ~* '^prefix\d*@.*'
UNION ALL SELECT id, email FROM users WHERE email ~* '^prefix\d*@.*'
ORDER BY created_at;
Red flag Multiple accounts with emails differing only by a digit suffix (e.g. foo3, foo33, foo333). Combine with check 2/3 for confirmation.

Check 5 — Does the payment method match the account name?

Stolen card / identity
SQL 
SELECT m.display_name, mpm.type, mpm.name, mpm.details
FROM merchants m JOIN merchant_payment_methods mpm ON mpm.merchant_id=m.id
WHERE m.id=$ID;
-- Same for user_payment_methods
Red flag Cardholder / UPI handle name does not match merchant display_name or KYC name. Could be normal (family card), or indicator of stolen identity.

Check 6 — What's their trade velocity pattern?

Structuring / burst
SQL 
SELECT DATE(created_at) AS day, COUNT(*) AS trades,
  SUM(fiat_amount) AS volume,
  COUNT(*) FILTER (WHERE fiat_amount < 10000) AS small_trades
FROM orders WHERE user_id=$ID OR merchant_id=$ID
GROUP BY 1 ORDER BY 1;
Red flag Many trades just below a regulatory threshold (e.g. ₹50,000 for KYC, ₹2,00,000 for STR). Sudden spike from 0 to high volume.

Check 7 — Who rated them, and is it suspicious?

Reputation manipulation
SQL 
SELECT r.rater_id, r.rating, r.created_at, o.id AS order_id, o.fiat_amount
FROM ratings r LEFT JOIN orders o ON o.id = r.order_id
WHERE r.rated_id=$ID
ORDER BY r.created_at;
-- Cross-check: does rater_id appear in device/IP cluster?
Red flag All ratings come from the same rater_id, or from raters in the same multi-account cluster. All ratings are 5★ on tiny trades.

Check 8 — Is their reputation score consistent?

Engine bug exploit
SQL 
SELECT total_score, tier, calculated_at,
  AGE(NOW(), calculated_at) AS staleness
FROM reputation_scores WHERE entity_id=$ID;
-- Check: does tier match score per the formula?
-- Check: was it updated recently?
Red flag Stale reputation (calculated_at > 7d old), or tier label inconsistent with total_score (current known bug — see merchant report).

Check 9 — Any disputes lost or fake-proof events?

Direct fraud
SQL 
SELECT d.*, o.fiat_amount FROM disputes d
JOIN orders o ON d.order_id = o.id
WHERE o.merchant_id=$ID OR o.user_id=$ID OR d.raiser_id=$ID
ORDER BY d.created_at DESC;
SELECT * FROM reputation_events
WHERE entity_id=$ID AND event_type IN ('dispute_lost','fake_proof','mark_paid_not_paid');
Red flag Any dispute resolved against them; any fake_proof event; cluster of mark_paid_not_paid penalties.

Check 10 — Did they file frivolous disputes?

Dispute abuse
SQL 
SELECT
  COUNT(*) FILTER (WHERE raiser_id=$ID) AS raised,
  COUNT(*) FILTER (WHERE raiser_id=$ID AND resolved_in_favor_of != 'user') AS lost
FROM disputes;
Red flag lost / raised > 50% with ≥3 raised. Core-api applies a frivolous-dispute trust penalty when this ratio exceeds 0.5.

Check 11 — Sudden behaviour change (account takeover)?

ATO
SQL 
SELECT created_at, ip_address, user_agent, device_info FROM sessions
WHERE entity_id=$ID ORDER BY created_at DESC LIMIT 20;
SELECT created_at FROM password_reset_tokens WHERE merchant_id=$ID
  OR user_id=$ID ORDER BY created_at DESC LIMIT 5;
Red flag New device + new IP after >30 days quiet, immediately followed by financial action. Recent password_reset_token use.

Check 12 — On-chain trades without app sessions?

Bypass / bot
SQL 
-- Time-correlate v2_trades vs sessions
SELECT v.created_at AS trade_time,
  (SELECT created_at FROM sessions s
    WHERE s.entity_id=$ID
    AND s.created_at < v.created_at
    ORDER BY created_at DESC LIMIT 1) AS prev_session
FROM v2_trades v
WHERE v.creator_pubkey=$WALLET OR v.counterparty_pubkey=$WALLET
ORDER BY v.created_at DESC LIMIT 10;
Red flag Many v2 trades when no recent session exists (means actor is hitting the chain directly, not via the app — possibly automated bot or non-account-holder access).

Check 13 — Risk engine signals (already computed)

Read existing
SQL 
SELECT * FROM risk_profiles WHERE entity_id=$ID;
SELECT * FROM risk_events WHERE entity_id=$ID ORDER BY created_at DESC;
SELECT * FROM security_alerts WHERE entity_id=$ID ORDER BY created_at DESC;
SELECT * FROM blacklist WHERE entity_id=$ID;
Red flag risk_level ∈ ('high','critical'); event_type IN ('multi_account_device','ip_cluster_detected','payment_retry_burst').

Check 14 — The "round-trip" wallet check

Circular trading
SQL 
-- Find chains: A → B → C → A within X hours
-- (approximation via counterparty graph on v2_trades)
WITH edges AS (
  SELECT creator_pubkey AS src, counterparty_pubkey AS dst, amount, created_at
  FROM v2_trades WHERE status='released'
)
SELECT e1.src, e1.dst, e2.dst AS three, e3.dst AS back_to_start
FROM edges e1
JOIN edges e2 ON e2.src = e1.dst AND e2.created_at BETWEEN e1.created_at AND e1.created_at + INTERVAL '24 hours'
JOIN edges e3 ON e3.src = e2.dst AND e3.dst = e1.src
WHERE e1.src=$WALLET LIMIT 50;
Red flag Funds return to origin within 24h. Strong indicator of layering / wash.

4. Detection Gaps

Things we currently can't see or don't see automatically:

GapWhy it mattersTo fix
Email similarity / pattern detection at signup Sequential-digit emails like zoopweb3/33/333 slip through individual email validation Add Levenshtein-distance check or numeric-suffix-stripped match against existing emails
Sanctions / blocked wallet feed No integration with OFAC / Chainalysis / TRM Labs sanctions lists Add periodic feed → blacklist table import
Cross-account on-chain analysis automation Wash-trade detection in core-api looks at one entity at a time — doesn't resolve counterparty wallets back to accounts to find the cluster Worker that builds the wallet ↔ account map and flags trades where both sides resolve to linked accounts
Reputation worker reliability Joe's reputation last updated 2026-05-11 — 9 days stale; on-chain volume not reflected Monitor + alert on calculated_at staleness; verify the activity-window query catches on-chain trades
Shill-rating detection No rule fires when a counterparty rates a trade 5★ within seconds — common shill pattern Rating velocity check + cross-cluster rater-rated overlap
Behavioural telemetry No click/typing/timing data; can't detect bot-like behaviour Out of scope for this DB — would need event-stream layer
Reverse-proxy IP loss 4 of 6 ip_logs for the Zooo merchant show ::1 (loopback) — real client IP not being captured Fix X-Forwarded-For / X-Real-IP handling in the API gateway
Real-name vs payment-method-name auto-match Currently a manual review; no rule fires when card name ≠ display name Add fuzzy-match flag at payment-method add time

5. Actions Available to Compliance

What you can do once an account is flagged, based on schema:

ActionWhereEffect
Suspend a merchant UPDATE merchants.status='suspended' + v_merchant_scoring.suspended_until Excluded from matching; can't accept new orders
Disable a user UPDATE users.kyc_status='blocked' (or similar — depends on auth gate) Login still works but actions blocked at API level
Blacklist INSERT INTO blacklist (entity_id, entity_type, reason, severity, is_active) Hard block on all entities of that id; checked by middleware
Force re-KYC UPDATE users.kyc_status='re_verify_required' Hold all actions until re-verified through UI flow
Revoke sessions UPDATE sessions SET is_revoked=true WHERE entity_id=… Forces re-login on every active session
Record manual reputation adjustment INSERT INTO reputation_events with event_type='manual_adjustment' Audit trail; reputation worker will pick up next cycle
Force escrow refund (open orders) atomicCancelWithRefund() via admin endpoint Closes order, returns crypto to escrow funder
Open dispute on their behalf INSERT INTO disputes with raiser_id='compliance' Locks order in disputed state for arbiter review

Recommended actions for this cluster

Immediate (today)

  1. Suspend all 6 named cluster accounts (Joe, Zooo, John, zoro, Alfa2, Olo) plus Alfa (dono4fire) and Hulk (dono.4fire) — 8 accounts total.
  2. Hold the $1,968 of on-chain volume — these trades may already be on-chain and irreversible, but flag any pending escrows.
  3. Blacklist device 8c7dea65… and IP 103.66.61.46 so future signups from this fingerprint are blocked.
  4. Reverse the shill 5★ rating on merchant John — and audit all of John's other ratings (he is the highest-activity merchant in the cluster).
  5. Reset reputation scores for the entire cluster to tier='risky' with manual_adjustment events.

Short term (this week)

  1. Add email pattern matching at signup — strip trailing digits + dots + plus addressing, compare against existing emails.
  2. Fix reverse proxy so real client IPs reach ip_logs (4 of 6 of Zooo's records are loopback).
  3. Investigate why the reputation worker stopped processing Joe on 2026-05-11.
  4. Add the wallet → account resolution as a periodic worker so on-chain wash-trade detection runs automatically.

Medium term

  1. Build a "cluster view" admin UI: given any account ID, show the full device/IP/email-pattern/wallet-counterparty graph.
  2. Add a shill-rating detector: rating posted <60s after order completion + rater previously linked by device/IP = flag.
  3. Integrate an external sanctions feed for wallet addresses.
