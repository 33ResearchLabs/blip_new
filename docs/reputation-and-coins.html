1. System overview

Blip runs two parallel, complementary scoring systems. They share the same audit ledger (blip_point_log) and the same daily worker, but serve different purposes.

System Range Default Direction Purpose Spendable?
Reputation 300–900 500 Can go up or down Trust signal; gates trade limits + perks No — derived
Blip Points (coins) 0–99,999 100 Earned only (lifetime monotonic); burnable for perks Activity reward + spend-to-unlock-limits Yes — see §4

Reputation reflects how trustworthy someone is right now. Blip Points reflect how engaged they have been. A user with high rep + low points = trusted but inactive. High points + low rep = active but unreliable. The product surfaces both together.

2. Reputation system (300–900)

CIBIL-style score. Computed internally from five weighted components, then linearly rebased from a 0–1000 internal range to the user-facing 300–900 range. Tier names map onto fixed score bands.

2.1 Component weights (internal 0–1000 input)

ComponentWeightWhat it captures
Review score 30%Average rating, count, recent trend
Execution score 25%Completion rate, speed, on-time, disputes/cancels
Volume score 15%Logarithmic weighted volume + counterparty diversity
Consistency score15%Account age, active days, longest inactive streak
Trust score 15%Dispute wins/losses, KYC level, verified flags

2.2 Rebase formula (internal → user-facing)

external_score = round(300 + (internal_score / 1000) * 600)
clamp to [300, 900]

So internal 0 → external 300 (Restricted), internal ~333 → 500 (New / default), internal 1000 → 900 (Platinum).

2.6 Hard gates

Independent of the score math, three hard floors apply:

  • Reliability gate — review score < threshold + ≥ N orders → forced Restricted, capped at 300.
  • Trust gate — trust score < threshold → forced Restricted.
  • Dispute-rate gate — > 15% disputes over the order count → forced Restricted, score capped.

3. Coin system (Blip Points)

Coins start at 100, earn through activity + quests, spend on trade-limit unlocks + perks. Hard ceiling of 99,999; a soft taper makes the last 50k progressively harder to earn so the system stays bounded.

3.2 Soft taper (anti-runaway)

Every credit is multiplied by a taper based on the actor's current TOTAL balance (locked + unlocked). Designed so even sustained activity can never reach 100k.

Balance bandEarn multiplier
0 – 50,0001.00×
50,000 – 80,0000.50×
80,000 – 95,0000.25×
95,000 – 99,9990.10×

3.3 Hard cap

HARD_CAP_COINS = 99_999. Any award that would exceed this is silently clipped to fit within headroom = HARD_CAP_COINS − (balance + locked).

3.4 Locked balances

Locks are time-released holds against the locked_blip_points column. Each lock has an unlocks_at timestamp; on reads they are swept lazily into the spendable balance. Used for:

  • Signup bonus (7-day hold — kills bot farming)
  • Unverified referrals (released when the referee trades)
  • Anti-abuse flagged credits (voided wholesale on confirmation)

3.5 Spend table (coin sinks)

Spending burns coins from the unlocked balance only. The audit ledger records the burn as a negative entry. Lifetime totals (used for reputation seed weights) are unaffected.

effective.perTrade = base.perTrade × rep_tier_multiplier × active_coin_unlock_multiplier

5. Anti-abuse playbook

Since Blip starts without KYC at the bottom tiers, the entire economy needs to be defensible against sybil farms. Each row below is a vector + the mitigation that's wired in.

VectorMitigation
Multi-account farming (sybil)Device-ID + fingerprint dedup — only one account per device earns signup/referral/task coins
Same payment-method farmingSalted hash of canonical PM identity; second-account credit zeroed
Wash-trade volume farmingCounterparty-diversity check — ≥ 3 unique counterparties required for volume coins/rep
Fast cancel-reaccept loopsSub-60s cancel pattern penalty; rate-limit cancellations to 5/day
Referral spamReferee must trade ≥ $50 within 30 days to credit referrer
Task replay (re-submitting same tweet)Server-side dedup on tweet ID, X user ID, Telegram user ID
Account flipping (sell aged account)Device + PM delta detection on sudden behaviour change → freeze rep+coins for review
Dispute farming (file then withdraw)Dispute filing alone costs −5 rep held; refunded on win, kept on withdraw
Coin laundering between accountsCoins are non-transferable, period (no transfer event type exists)
Bot signupsBehavioural signal (signup time < 3s, no mouse entropy) → coins locked, never credited
KYC voucher resaleKYC bound to actor_id, not transferable, single-use
Streak gaming via timezone hoppingStreaks computed against the account's first-detected timezone, locked 90d
Mass referrals from bought emailsIPQS check on referee signup; disposable email / VOIP phone → referral credit voided
Grinding past 50kSoft taper (1.0 → 0.5 → 0.25 → 0.10) requires sustained genuine volume to exceed 85k

9. Scaling decisions

Designed for 100M users from day 1, but with deferrable optimisations for when row counts justify them. Hot reads are O(1) via denormalised columns; hot writes are protected by indexed UPSERTs.

ConcernResolution
Rep recompute per user per hour at 100M scale (~27k/s)Drop hourly worker. Recompute on event (trade complete, dispute, task verify) + lazy daily for dormant
Append-only ledger growth (blip_point_log)Composite indexes + BRIN on created_at. Native monthly partitioning deferred to a future migration once row count crosses 50M
Hot balance readsDenormalized blip_points on users/merchants. Redis caching layer added when read-rate > 5k/s
Cap-state lookups on every earnComposite index (actor, event, period_start) + partial index for current-period only
Race conditions on credit/debittransaction() + FOR UPDATE on actor row; conditional WHERE balance >= points for burns
Locked-coin unlockLazy on read — no scheduled job. Always one extra SELECT before balance, gated by partial index WHERE released_at IS NULL
Coin transfer between accountsDisabled at schema level — no TRANSFER event type. Cannot launder coins
Worker memory under 100M usersProcess actors in batches of 500 with Promise.all. Worker process holds bounded heap
UI fan-out on dashboard loadOne /api/reputation/me + one /api/coins/me call per page mount, no polling

10. How to amend values

Every tunable lives in one of three places. Find the right one before editing — there's exactly one source of truth per value.

10.1 Coin amounts

Single source: settle/src/lib/coins/awards.ts, exported AMOUNTS constant. Per-event caps: CAP_POLICIES in settle/src/lib/coins/economy.ts. To change "5-star is worth 10 coins" or the daily trade cap, edit those constants. No migration needed — values are applied on next worker tick.

10.2 Hard cap and taper bands

HARD_CAP_COINS and softTaperMultiplier() in settle/src/lib/coins/economy.ts. Editing these affects new earns only; existing balances are unaffected.

10.3 Reputation tier thresholds + multipliers

Tier ranges: TIER_THRESHOLDS in settle/src/lib/reputation/types.ts. Daily/per-trade multipliers per tier: REP_TIER_MULTIPLIER in settle/src/lib/coins/limits.ts. Display labels: TIER_INFO in types.ts.

10.4 Component weights inside the rep score

WEIGHTS in settle/src/lib/reputation/calculator.ts controls how reviews / execution / volume / consistency / trust combine. Changing these takes effect on the next recompute (daily worker pass).

10.5 Trade-limit tier prices

COIN_LIMIT_TIERS in settle/src/lib/coins/limits.ts. Prices, daily / per-trade USD caps, KYC requirement per tier all live there. Currently active unlocks are honoured at their original price — only NEW purchases reflect the change.

10.6 Backfill of legacy accounts

Use a NEW migration file (don't edit migration 134). The pattern from 134 is: 

INSERT INTO blip_point_log (actor_id, actor_type, event, bonus_points, source_ref, metadata)
SELECT u.id, 'user', 'MANUAL_CREDIT', N, 'unique-source-ref-for-this-backfill', '{"reason":"..."}'::jsonb
FROM users u
WHERE ...condition...
ON CONFLICT DO NOTHING;

UPDATE users SET blip_points = GREATEST(COALESCE(blip_points,0), N), updated_at = NOW()
WHERE ...condition...;

Always use a fresh source_ref per backfill — the idempotency index prevents re-applying the same one.

SymptomLikely causeFix
