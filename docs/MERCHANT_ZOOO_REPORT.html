  • IP cluster detected (MEDIUM, 6 occurrences) — signups/logins from IPs (103.66.61.46 and ::1) that have been used by 7–9 unique users.

⚠️ Reputation tier inconsistency (likely a bug)

The merchant has 0 orders, 0 ratings, and 0 trades, yet their stored reputation row reads:

total_score = 400, tier = "diamond"

Per settle/src/lib/reputation/types.ts, score 400 should map to silver, and diamond requires ≥900. Two probable causes:

  1. Core-api stores components on the 0–1000 scale (volume=500, consistency=300, trust=700) while Settle's schema expects 0–100. Schema drift between the two engines.
  2. The tier classifier produced "diamond" from a different calculation path (possibly an old default). Worth verifying in apps/core-api/src/reputation/scoring.ts:assignTier.

The denormalised merchants.reputation_score = 400 with merchants.reputation_tier = "diamond" is what the matching engine sees — they will be ranked far higher than they should be.

⚠️ Trust level vs reputation tier contradiction

The v_merchant_scoring view reports trust_level = "probation" (consistent with high risk + zero activity), but the reputation engine says tier = "diamond". These two systems disagree about whether this merchant is trustworthy.
