Order Creation POST /v1/orders

A user picks a merchant's offer and creates a trade. The system deducts liquidity first (short row lock on the offer), then inserts the order row.

Expiry: expires_at = now() + 15 minutes (configurable, clamped 1–1440 min).

Idempotency: Header idempotency-key scoped by actor prevents duplicate creation on retries.

apps/core-api/src/routes/orderCreate.ts:185-208 TypeScript 
// 1. Deduct offer liquidity FIRST (short row lock, auto-commit)
const deducted = await dbQuery(
  'UPDATE merchant_offers SET available_amount = available_amount - $1
   WHERE id = $2 AND available_amount >= $1 RETURNING id',
  [data.crypto_amount, data.offer_id]
);
if (deducted.length === 0) {
  return reply.status(409).send({ error: 'Insufficient offer liquidity' });
}

// 2. Insert order row
const rows = await dbQuery(
  `INSERT INTO orders (..., expires_at) VALUES (..., now() + interval '15 min') RETURNING *`,
  values
);

// 3. Emit event -> all 4 listeners fire
orderBus.emitOrderEvent({
  event: ORDER_EVENT.CREATED,
  orderId: order.id,
  newStatus: order.status,   // 'pending' or 'escrowed'
});
⚠
Liquidity deduct and order insert are NOT in a transaction. A crash between them could leak liquidity. Mitigated by idempotency, but worth monitoring.
DB orders row inserted (status: pending) • merchant_offers.available_amount decremented
Pusher order:created → private-merchants-global (all merchants see new order)

export function pusherNotifyOrderStatus(payload) {
  const channels = [
    getOrderChannel(payload.orderId),     // private-order-{id}
    getUserChannel(payload.userId),       // private-user-{userId}
    getMerchantChannel(payload.merchantId), // private-merchant-{mid}
  ];

  // Broadcast to all merchants for pool-affecting statuses
  if (['accepted', 'cancelled', 'expired'].includes(payload.status)) {
    channels.push(getAllMerchantsChannel());
  }

  triggerEvent(channels, 'order:status-updated', {
    orderId, status, minimal_status,
    order_version,  // frontend uses this to reject stale updates
    previousStatus, updatedAt,
    data: fullOrderSnapshot,
  });
}
💡
Stale update protection: The payload includes order_version. The frontend can compare this with its local version and discard events with lower versions to avoid flicker.

Queue System (BullMQ)

📦
Receipt Queue Only

BullMQ is used only for receipt processing. All other background tasks (expiry, bumps, prices) use polling workers.

Why? Receipt creation involves 3–4 DB queries. Running this synchronously in the API response would add ~100ms latency to every order acceptance.

Risks & Important Notes

🚨
Non-atomic order creation: Liquidity deduction and order insert are two separate queries (not a transaction). A crash between them could leak liquidity from merchant_offers. Idempotency keys mitigate retries but don't cover mid-query crashes.
⚠
Receipt 2s delay: The BullMQ CREATE_RECEIPT job has a 2-second delay. Users opening chat immediately after acceptance may not see the receipt card yet. Plus retry backoff (2s → 4s → 8s) on failure.
⚠
payment_sent stall attack: Since payment_sent orders never expire, a malicious buyer could mark payment_sent without actually sending fiat, stalling the order indefinitely. Mitigated by unhappyPathWorker and the dispute system, but worth monitoring.
⚠
Redis down = no receipts: If Redis (Upstash) is unavailable, the receipt queue silently no-ops. Receipts will be missing until Redis recovers. There is no catch-up mechanism to retroactively create missing receipts.
💡
Forward-only receipt guard: If BullMQ delivers events out of order (e.g., a slow escrowed update arrives after payment_sent), the STATUS_PRIORITY guard silently drops the stale update. This is by design, not a bug.
💪
Strong concurrency safety: Stored procedures (accept_order_v1, escrow_order_v1), optimistic locking (order_version), FOR UPDATE SKIP LOCKED in the expiry worker, and ON CONFLICT DO NOTHING for receipts make this system highly resilient to race conditions.
