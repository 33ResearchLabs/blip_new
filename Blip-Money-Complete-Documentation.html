1. Trade Flow

GOLDEN RULE: The seller ALWAYS locks AND releases the escrow. Fiat is always sent offchain (bank transfer / cash). Crypto goes through on-chain escrow.

BUY Order Flow (User wants to buy crypto)

1. BUYER places order: "I want 100 USDC, paying 367 AED"
2. SELLER accepts the order (becomes counterparty)
3. SELLER locks 100 USDC into escrow (balance: 10000 → 9900)
4. BUYER receives seller's bank details
5. BUYER sends 367 AED via bank transfer (offchain)
6. BUYER clicks "Payment Sent" in app
7. SELLER confirms AED received in bank
8. SELLER releases 100 USDC from escrow to buyer
9. ✔ Order completed. Buyer: +100 USDC. Seller: +367 AED.

SELL Order Flow (User wants to sell crypto)

1. SELLER locks crypto into escrow BEFORE order goes live (pre-locked)
2. Order appears on marketplace with escrow already locked
3. BUYER accepts the order
4. BUYER sends fiat via bank transfer (offchain)
5. BUYER clicks "Payment Sent"
6. SELLER confirms fiat received
7. SELLER releases crypto from escrow to buyer
8. ✔ Order completed.

M2M Flow (Merchant-to-Merchant)

M2M BUY (Merchant A buys from Merchant B):
1. A creates BUY order (stored inverted as type="sell")
2. B accepts → becomes seller, merchant_id reassigned to B
3. B locks escrow → B balance decreases
4. A sends fiat, marks "payment sent"
5. B confirms fiat received, releases escrow → A gets crypto
M2M SELL (Merchant A sells to Merchant B):
1. A creates SELL order (stored inverted as type="buy")
2. B accepts → buyer_merchant_id set to B
3. A locks escrow → A balance decreases
4. B sends fiat, marks "payment sent"
5. A confirms, releases escrow → B gets crypto
Type Inversion: When a merchant creates a BUY order, it's stored as type=sell in DB (user perspective). When a merchant creates a SELL order, it's stored as type=buy in DB.

2. Order Statuses & State Machine

Status Flow (Happy Path)

pending→ accepted→ escrow_pending→ escrowed→ payment_sent→ payment_confirmed→ completed ✔

All 12 Statuses

StatusDescriptionTimeoutWho Can Act
pendingOrder created, awaiting acceptance15 minAny merchant can accept; creator can cancel
acceptedCounterparty accepted, awaiting escrow15 minSeller locks escrow
escrow_pendingEscrow being verified on-chain—System only
escrowedCrypto locked in escrow120 minBuyer sends fiat; either can cancel/dispute
payment_pendingM2M intermediate state—System
payment_sentBuyer marked fiat sent15 minSeller confirms receipt
payment_confirmedSeller confirmed fiat received (transient!)—Seller releases escrow
releasingEscrow release in progress—System
completedOrder done, funds settled—Terminal — no actions
cancelledOrder cancelled—Terminal — refund if escrowed
disputedPayment dispute raised24 hrCompliance resolves; auto-refund on timeout
expiredOrder timed out—Terminal
payment_confirmed is TRANSIENT. isTransientStatus('payment_confirmed') returns true. It normalizes to payment_sent. Core-api rejects PATCH to transient statuses. Release directly from payment_sent.

Cancellation Rules

WhenWhoEffect
Before escrow (pending/accepted)Either partyClean cancel, no refund needed
After escrow (escrowed/payment_sent)Mutual agreementEscrow refunded to seller
After escrow, no agreementEither partyBecomes dispute

Extension System

  • Max 3 extensions per order
  • Extension 1: +15 min | Extension 2: +30 min | Extension 3: +60 min
  • Counterparty must approve the extension request

Core-API Authentication

All core-api endpoints (except /health and /debug/*) require x-core-api-secret header matching CORE_API_SECRET env var. Actor identity headers (x-actor-type, x-actor-id) are HMAC-SHA256 signed to prevent forgery.

8. Escrow System

Escrow Lock (Atomic DB Transaction)

1. Lock order row (SELECT FOR UPDATE)
2. Validate order not already escrowed
3. Validate state transition allowed
4. Determine payer (seller's account)
5. Lock payer's balance row (FOR UPDATE)
6. Check sufficient balance ≥ order amount
7. Deduct balance atomically
8. Update order: escrow_tx_hash, status='escrowed', escrow_debited_entity_id
9. Insert ledger entry (escrow_lock, negative amount)
10. Insert merchant_transactions record
11. Insert order_events audit record

Escrow Release (Atomic DB Transaction)

1. Validate escrow is locked (escrow_tx_hash exists)
2. Validate actor is seller (authorized to release)
3. ALREADY_RELEASED guard — check release_tx_hash IS NULL
4. INVALID_STATUS guard — only from payment_sent/escrowed/payment_confirmed
5. Determine payee (buyer's account)
6. Lock payee's balance row (FOR UPDATE)
7. Credit buyer balance
8. Update order: release_tx_hash, status='completed'
9. Insert ledger entry (escrow_release, positive amount)
10. Notify both parties via Pusher

Escrow Refund (on cancel/timeout)

1. Determine who locked escrow (escrow_debited_entity_id)
2. Refund that account (credit balance)
3. Insert refund ledger entry
4. Update order status to cancelled

9. Corridor & Conversions

Corridor LP Matching

POST /v1/corridor/match — auto-matches the cheapest available LP, locks buyer's sAED (fiat + fee), creates fulfillment record with 30-minute deadline.

Currency Conversion

Atomic USDT ↔ sINR/sAED conversion with floor rounding (prevents value creation), idempotency key support, exposure limits, and full ledger audit trail.

  • USDT → sINR: Input in micro-USDT (1,000,000 = 1 USDT), output in fils (100 = 1 AED)
  • sINR → USDT: Input in fils, output in micro-USDT
  • Default exposure limit: 90% of USDT backing
