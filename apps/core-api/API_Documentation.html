A Authentication

All endpoints except /health and /debug/* require authentication via shared secret header. Actor identity headers provide HMAC-signed authorization.

Required Headers
HeaderTypeRequiredDescription
x-core-api-secretstringRequiredShared secret matching CORE_API_SECRET env var
x-actor-typestringOptionaluser | merchant | system
x-actor-idstringOptionalUUID of the acting entity
x-actor-signaturestringIf actor headersHMAC-SHA256 of {actor_type}:{actor_id} using the shared secret
ⓘ
Dev mode: If CORE_API_SECRET is not set and NODE_ENV != production, auth is disabled entirely.
Error Responses
{ "success": false, "error": "Unauthorized: invalid or missing x-core-api-secret" }  // 401
{ "success": false, "error": "Unauthorized: missing actor signature" }                // 401
{ "success": false, "error": "Unauthorized: invalid actor signature" }                 // 401

L Order Lifecycle

The order state machine governs all transitions. Settle proxy handles user-facing validation; core-api handles DB writes and broadcasts.

created, waiting for merchant
➤

➤

crypto locked on-chain
➤

➤
payment_confirmed
seller confirms receipt
➤

Side paths: cancelled • expired • disputed

PATCH /v1/orders/:id Transition Order Status

The main status transition endpoint. Handles the full order state machine including accept, escrow, payment flow, completion, and cancellation. Validates transitions, prevents self-acceptance, handles escrow refunds atomically.

Request Body
FieldTypeReqDescription
statusOrderStatusReqTarget status: accepted, escrowed, payment_pending, payment_sent, payment_confirmed, completed, cancelled
actor_type"user" | "merchant"ReqWho is performing the action
actor_iduuidReqUUID of actor
reasonstringOptCancellation reason
acceptor_wallet_addressstringOptWallet address when accepting
metadataobjectOptExtra data (e.g. is_m2m)
Key Behaviors
ⓘ
accepted: Merchant claims order. If merchant != original merchant_id, becomes the new merchant. Sets expires_at = NOW() + 120 min. Acceptor wallet stored if provided.
ⓘ
payment_sent: Fast path — simple status flip from escrowed only, no transaction needed.
⚠
cancelled with escrow: Triggers atomicCancelWithRefund — restores balance, restores offer liquidity, verifies refund invariants.
⚠
completed: Cannot complete if escrow_tx_hash exists but release_tx_hash is missing. Use POST /v1/orders/:id/events with event_type: "release" instead.
Response — 200
{ "success": true, "data": { /* full order with minimal_status */ } }
Error Responses
400 Invalid transition, self-acceptance blocked, transient status, missing fields
404 Order not found
500 Refund invariant failed (critical)

POST /v1/orders/:id/events Release / Refund Escrow

Finalizes an escrowed order by releasing crypto to buyer or refunding to seller. Requires actor identity headers.

Headers Required
HeaderDescription
x-actor-typeuser | merchant | system
x-actor-idUUID of the actor
Request Body
FieldTypeReqDescription
event_type"release" | "refund"ReqRelease (complete) or refund (cancel)
tx_hashstringFor releaseOn-chain release transaction hash
reasonstringFor refundReason for refund
✔
Release: Sets status = completed, stores release_tx_hash, credits buyer balance (mock mode). Runs invariant verification. Only allowed from payment_confirmed, releasing, escrowed, payment_sent, payment_pending.
⚠
Refund: Atomic cancel with escrow refund. Restores balance and offer liquidity.
400 Already released, invalid status, missing tx_hash
401 Missing actor headers
403 Not authorized to release
